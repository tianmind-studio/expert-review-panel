# 小程序代码 / 微信生态 · 评审专家库

## 适用场景

微信小程序、小游戏、公众号内嵌小程序、云开发项目、Taro / uni-app / mpvue 等编译到 `mp-weixin` 的代码审查；也适用于提审前技术预审、隐私权限整改、云函数安全审计、支付与内容安全相关变更。

典型输入包括：

- `app.json`、`project.config.json`、`sitemap.json`
- `pages/**`、`components/**`、`utils/**`
- `cloudfunctions/**`、`miniprogram/**`
- Taro / uni-app / mpvue 源码与编译后的微信小程序产物
- 小程序提审备注、服务类目、隐私接口清单、支付/订阅消息/分享/客服消息相关代码

## 使用前提醒

微信开放平台规则、隐私接口要求、服务类目、备案、支付与内容安全要求会变化。做具体规则判断前，应刷新官方文档；本专家库只提供代码审查框架，不保证平台审核结果。

## 专家阵容（从中挑 4-7 位组队）

### 1. 小程序架构负责人（Miniapp Architect M）

**审查重点**：页面/组件边界、分包策略、启动链路、全局状态、路由设计、跨端框架编译产物、插件/第三方 SDK 的边界，以及小程序运行时限制下的演进路径。

**典型追问**：首屏必须加载哪些包？哪些页面应该独立分包？`app.js` 是否承担了过多业务状态？跨端框架生成的代码有没有绕过小程序限制或造成调试盲区？

### 2. 微信平台合规审查员（Platform Reviewer W）

**审查重点**：服务类目与真实功能一致性、提审路径可达性、隐私接口调用时机、强制登录、诱导分享/关注/下载、外部跳转、客服消息、订阅消息、备案和资质风险。

**典型追问**：审核员从首页两次点击内能到达核心功能吗？为什么一打开就要登录或授权？这个跳转是不是在把小程序体验导向 H5 / App / 公众号？提审备注有没有交代测试账号和关键路径？

### 3. 云函数 / 后端安全审计员（Cloud Security S）

**审查重点**：`cloud.getWXContext()` 使用、`openid` / `unionid` 信任边界、云函数权限、数据库规则、越权读写、API 鉴权、密钥管理、云调用权限、服务端校验。

**典型追问**：有没有信任前端传来的 `openid`？数据库读写权限是不是默认全开？云函数能不能被任意用户传入别人的对象 ID 后越权修改？AppSecret 是否出现在前端包、日志或仓库里？

### 4. 隐私与数据最小化审查员（Privacy Lead P）

**审查重点**：用户信息收集目的、隐私保护指引覆盖、权限弹窗时机、手机号/位置/头像昵称等敏感信息处理、缓存与本地存储、日志脱敏、数据删除和投诉路径。

**典型追问**：这个页面真的需要手机号/定位/头像昵称吗？隐私接口调用前是否有场景化说明？`wx.setStorageSync` 是否长期保存了可识别个人信息？日志是否会打印 token、手机号或身份证号？

### 5. 内容安全 / UGC 审查员（Content Safety C）

**审查重点**：用户发布内容、评论、昵称头像、图片/音视频上传、AI 生成内容、搜索关键词、举报与下架路径、内容安全接口接入和失败兜底。

**典型追问**：所有用户可见内容在发布前或展示前经过安全检查了吗？安全接口失败时是放行、阻断还是人工复核？是否有举报、删除、封禁和审计记录？

### 6. 性能与体验工程师（Performance Engineer R）

**审查重点**：首屏启动、分包加载、图片与媒体体积、setData 频率、长列表渲染、网络超时、弱网重试、缓存策略、错误态/空态、基础库兼容性。

**典型追问**：首屏白屏多久？列表每次 `setData` 的 payload 多大？失败态有没有用户可理解的反馈？弱网下会不会重复提交订单或发布内容？

### 7. 支付 / 交易风控审查员（Payment Risk F）

**审查重点**：微信支付、虚拟商品、退款、订单幂等、金额与库存校验、支付回调验签、发货/履约、投诉与售后、诱导外部付款。

**典型追问**：金额是否只以前端传参为准？支付回调有没有验签和幂等？虚拟内容或会员权益是否符合小程序支付边界？是否把用户引导到个人微信、H5 或其他渠道付款？

### 8. 魔鬼代言人（Devil's Advocate）

**审查重点**：挑战“这个只是小程序，风险不大”的默认放松。

**典型追问**：如果审核员、恶意用户、普通新用户各自第一次打开这个版本，最先失败在哪里？如果这套代码明天被 10 倍用户量或一次公开传播打中，哪个假设会先塌？

## 小程序代码常见致命缺陷（P0 候选库）

- **信任前端身份**：云函数或后端接口信任前端传来的 `openid`、`userId`、角色、价格、订单状态。
- **权限全开**：云数据库规则、对象存储、管理接口缺少用户级/租户级/对象级权限校验。
- **强制登录或过早授权**：打开首页即要求头像昵称、手机号、定位等，与当前功能场景不匹配。
- **隐私指引不一致**：代码调用的隐私接口、收集字段、使用目的没有在隐私保护指引或页面说明中体现。
- **UGC 无内容安全**：评论、帖子、昵称头像、上传图片、AI 输出等直接发布或展示，没有安全检测和人工处理路径。
- **诱导或绕行平台规则**：诱导分享/关注/下载，绕到 H5/App/个人微信付款，隐藏功能给审核员看不到。
- **支付风控缺失**：金额以前端为准、支付回调无验签/幂等、退款无审核、订单状态可被重复推进。
- **提审路径不可达**：核心功能藏得太深、依赖无测试账号、页面空白或只有占位内容，审核员无法复现真实价值。
- **前端泄密**：AppSecret、云开发密钥、第三方 API Key、内部管理接口出现在小程序包、日志或公开仓库。
- **弱网和重复提交失控**：无 loading/disabled/幂等 token，用户连续点击导致重复发布、重复支付、重复下单。

## 严苛度校准

- **准备正式提审 / 已被驳回后复审**：平台合规、隐私、内容安全、核心功能可达性按高压标准处理；任一 P0 即 NO-GO。
- **涉及支付、医疗、教育、金融、政务、未成年人或大量 UGC**：默认提高一个严苛等级；安全、隐私、资质问题不允许拖后。
- **内部测试版 / 原型**：可以接受体验 P2 和局部功能缺口，但身份、权限、密钥、支付、隐私 P0 不可放宽。
- **学习项目 / 课程作业**：指出生产级风险，同时把修改方向写成可学习的具体步骤；不要承诺可直接提审。
